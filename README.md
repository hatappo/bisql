# Bisql

Bisql (pronunce `bάɪsɪkl` 🚲️ ) is 2-way-sql ORM library for Clojure.

## Development

### Prerequisites

```sh
brew install clojure/tools/clojure
# If you don't use Mac, see https://clojure.org/guides/install_clojure

brew install borkdude/brew/babashka
# If you don't use Mac, see https://github.com/babashka/babashka#installation
```

### Setup

#### Setup Clojure

```sh
clj -P
```

#### Setup PostgreSQL

```sh
docker compose up -d
bb test-db
```

Connection settings:

- host: `localhost`
- port: `5432`
- database: `bisql_dev`
- user: `bisql`
- password: `bisql`

Connect with `psql`:

```sh
psql -h localhost -p 5432 -U bisql -d bisql_dev
```

Quick checks:

```sql
\dn+

\d

SELECT * FROM users;
SELECT * FROM orders;
```

### Generate CRUD SQL Files

Generate CRUD templates from the local PostgreSQL schema and group them into one SQL file per table:

```clj
(require '[bisql.core :as bisql]
         '[next.jdbc :as jdbc])

(def datasource
  (jdbc/get-datasource {:dbtype "postgresql"
                        :host "localhost"
                        :port 5432
                        :dbname "bisql_dev"
                        :user "bisql"
                        :password "bisql"}))

(-> (bisql/generate-crud datasource {:schema "public"})
    (bisql/write-crud-files! {:output-root "src/sql"}))

(-> (bisql/generate-crud datasource {:schema "public"})
    (bisql/write-crud-query-namespaces! {:output-root "src/sql"}))
```

This writes files such as:

- `src/sql/postgresql/public/users/users-crud.sql`
- `src/sql/postgresql/public/orders/orders-crud.sql`
- `src/sql/postgresql/public/users.clj`
- `src/sql/postgresql/public/orders.clj`

Each generated namespace file loads the matching SQL directory, for example:

```clj
(ns sql.postgresql.public.users
  (:require [bisql.core :as bisql]))

(bisql/defquery)
```

SQL templates are resolved from the classpath under the logical `sql` base path.
That means they can live under `src/sql/`, `resources/sql/`, or any other classpath root
that exposes `sql/...`.

Or run the bundled task:

```sh
bb gen-config
```

This writes a `bisql.edn` template:

```clojure
{:db {
      ;; :dbtype "postgresql"
      ;; :host "localhost"
      ;; :port 5432
      ;; :dbname "bisql_dev"
      ;; :user "bisql"
      ;; :password "bisql"
      }
 :generate {
            ;; :schema "public"
            ;; :base-dir "src/sql"
            }}
```

コメントは既定値の例です。ファイルを編集しなくても、CLI は既定値で動作します。

Then generate CRUD SQL files:

```sh
bb gen-crud
```

Or use the CLI directly:

```sh
clojure -M -m bisql.cli gen-config
clojure -M -m bisql.cli gen-crud --config bisql.edn
```

If you also want the namespace wrapper files generated:

```sh
bb gen-ns
```

```sh
clojure -M -m bisql.cli gen-ns --config bisql.edn
```

`bb gen-crud` and `bb gen-ns` are thin wrappers around `bisql.cli`.
The same options can also be passed through environment variables such as
`BISQL_HOST`, `BISQL_PORT`, `BISQL_DBNAME`, `BISQL_USER`, `BISQL_PASSWORD`,
`BISQL_SCHEMA`, `BISQL_BASE_DIR`, `BISQL_DBTYPE`, and `BISQL_CONFIG`.

The precedence order is CLI options > environment variables > config file > defaults.

## Define Query Functions

You can define rendering functions directly from SQL files.

```clj
(ns app.user-queries
  (:require [bisql.core :as bisql]))

(bisql/defrender)
(bisql/defrender "/sql/postgresql/public/users/get-by-id.sql")

(get-by-id {:id 42})
;; => {:sql "SELECT * FROM users WHERE id = ?"
;;     :params [42]
;;     :query-name "get-by-id"
;;     :meta {}}
```

`(bisql/defrender)` loads every `.sql` file under the default `sql` base path
recursively and defines every query it finds based on the current namespace path.

You can also pass a relative or absolute path.

- A relative path is resolved under the current namespace-derived directory.
- A path starting with `/` is resolved from the classpath root.

```clj
(bisql/defrender "admin")
(bisql/defrender "/sql/postgresql/public/users")

(get-by-id {:id 42})
(list-by-status {:status "active" :limit 100})
```

The same `defrender` also works for multi-query files:

```clj
(bisql/defrender "/sql/example-multi-queries.sql")

(find-user-by-id {:id 42})
(find-user-by-email {:email "user@example.com"})
```

`defrender` resolves the function name from `query-name`.
The `query-name` priority is:

1. `/*:name */`
2. the SQL file name itself

For example, these define `get-by-id`, `find-user-by-id`, and `find-user-by-email`.
With no arguments, `defrender` processes all `.sql` files under the current
namespace-derived directory recursively. When a directory is passed, it
processes all `.sql` files under that directory recursively in sorted path order.

Execution adapters live under adapter namespaces such as `bisql.adapter.next-jdbc`,
so the `core` namespace can stay focused on loading, analyzing, rendering, and function generation.

```clj
(ns app.user-queries
  (:require [bisql.core :as bisql]
            [bisql.adapter.next-jdbc :as bisql.jdbc]))

(bisql/defquery "/sql/postgresql/public/users/users-crud.sql")

(bisql.jdbc/exec! datasource get-by-id {:id 42})
```

`bisql.adapter.next-jdbc/exec!` chooses `next.jdbc/execute-one!` or
`next.jdbc/execute!` from the query function metadata's `:cardinality` value.
If `:cardinality` is not specified, it defaults to `:many`.
